适用场景

快速开始

如果你只想尽快跑起来，先从当前仓库里的 Docker 方式开始

仓库当前文档建议优先使用 Docker 部署。下面这组步骤适合第一次快速启动，再根据你的音乐目录、配置文件和端口需求继续调整。

  1. 准备配置文件，把 config.sample 复制或改名为 supysonic.conf。
  2. 构建镜像。
  3. 挂载音乐目录与配置文件，启动容器。
  4. 按你的环境继续补充用户、媒体库和扫描配置。
git clone https://github.com/jsdfhasuh/supysonic
cd supysonic
cp config.sample supysonic.conf

docker build -t supysonic .

docker run -d \
  -p 5000:5000 \
  -v /path/to/your/music:/music \
  -v /path/to/your/supysonic.conf:/app/supysonic.conf \
  supysonic

FAQ

先回答几个最容易在第一次了解项目时出现的问题

这是一个自托管项目吗？
是。页面中的所有描述都围绕自托管场景展开，当前仓库也提供了配置文件、守护进程、扫描器和 Docker 部署方式。
可以直接用 Docker 部署吗？
可以。当前 README 明确建议优先通过 Docker 部署，仓库中也提供了 Dockerfile 与启动脚本。
它和 Subsonic 生态是什么关系？
当前项目建立在 Subsonic API 之上，源自 Supysonic，并在此基础上继续扩展服务端能力与 Emosonic 相关功能。
除了播放之外，它还重点解决什么问题？
重点还包括媒体库维护，例如封面补全、年份修复、艺术家资料补全、local nfo 组织，以及扫描与 watcher 驱动的目录维护流程。
项目源码和扩展仓库在哪里看？
页面里的两个 GitHub 按钮分别直达 jsdfhasuh/supysonic 和 jsdfhasuh/emosonic，适合查看源码、README 和提交记录。
